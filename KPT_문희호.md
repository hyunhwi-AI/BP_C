## Keep

### 요구사항 정의 & 문서화

- README를 보고 바로 구현에 들어가지 않고 PRD, TEST_PLAN, RED ID를 먼저 정리하면서 요구사항을 테스트 가능한 단위로 분해하는 과정을 경험할 수 있었다.
- Mom Test를 통해 "무엇을 만들 것인가"보다 "실제로 사용자가 어떤 문제를 겪고 있는가"를 먼저 정의하는 과정의 중요성을 이해할 수 있었다.
- RED ID, Canonical TC, Requirement ID를 연결하면서 요구사항부터 테스트, 코드까지 추적성을 유지하는 방법을 경험할 수 있었다.
- PRD → TEST_PLAN → RED → GREEN → REFACTOR 순서로 진행하면서 개발 과정 전반을 구조적으로 관리하는 방법을 경험할 수 있었다.

### TDD & 테스트 설계

- Dual-Track TDD를 적용하면서 Logic Track과 UI Track을 분리해 생각할 수 있었고, 각 테스트가 어떤 책임을 검증하는지 명확하게 구분할 수 있었다.
- RED Skeleton → GREEN → REFACTOR 흐름을 직접 경험하면서 구현보다 테스트와 설계가 먼저라는 TDD의 핵심 개념을 이해할 수 있었다.
- 테스트를 먼저 작성하다 보니 구현이 요구사항 범위를 벗어나지 않도록 자연스럽게 통제할 수 있었다.
- Golden Master를 활용하여 출력 포맷의 변경 여부를 검증할 수 있었고, 기능 검증과 출력 검증을 분리하는 방법을 학습할 수 있었다.

### 아키텍처 & 설계

- ECB(Entity-Control-Boundary) 구조를 적용하면서 입력, 흐름 제어, 비즈니스 로직의 책임을 분리하는 방법을 학습할 수 있었다.

### AI 활용

- Cursor Ask와 Agent를 역할에 맞게 구분하여 사용하면서 AI를 단순 코드 생성기가 아닌 설계·검토 파트너로 활용할 수 있었다.
- AI에게 바로 구현을 요청하기보다 PRD와 TEST_PLAN을 먼저 작성하도록 유도했을 때 더 안정적이고 일관된 결과를 얻을 수 있었다.

## Problem

### 개념 학습 & 이해

- C2C, ECB, Mom Test, Dual-Track TDD, Golden Master 등 처음 접하는 개념이 많아 각 개념의 역할과 관계를 이해하는 데 예상보다 많은 시간이 소요되었다.
- RED Skeleton과 Executable RED의 차이를 이해하는 데 시간이 걸렸다. RED 단계가 끝났다고 생각했지만 GREEN 단계에서 테스트가 실제로 실행되지 않는 문제를 경험하면서 두 개념의 차이를 명확히 인지하게 되었다.
- Golden Master는 출력 변경 감지에는 효과적이지만, 처음에는 일반 테스트와 어떤 역할 차이가 있는지 이해하는 데 시간이 필요했다.
- Command, Skill, Rule, Hook 등 Cursor 기반 개발 환경의 구성 요소가 많아 처음에는 각각의 역할과 사용 시점을 구분하는 것이 어려웠다.

### 문서화 & 요구사항 분해

- README의 요구사항을 PRD로 정리하고, 다시 TEST_PLAN과 RED ID로 연결하는 과정에서 어떤 수준까지 요구사항을 분해해야 하는지 판단하기 어려웠다.
- PRD의 Requirement, Canonical TC, RED ID를 연결하는 과정에서 추적성은 높아졌지만 문서 작성에 많은 시간이 소요되었다.
- 요구사항을 구현하기보다 요구사항을 테스트 가능한 형태로 정의하는 과정이 더 어렵다는 점을 경험했다.

### TDD 실천 & 설계 판단

- 하나의 프롬프트로 여러 테스트를 동시에 GREEN 처리하려고 하면서 TDD의 "작은 단위 반복" 원칙이 흐려질 수 있었다.
- ECB 구조를 적용하면서 boundary, control, entity 중 어느 계층이 책임져야 하는지 판단하는 과정이 생각보다 쉽지 않았다.
- 리팩터링 단계에서 기능 개선과 구조 개선의 경계를 구분하는 것이 어려웠다. 테스트를 통과시키는 수정과 코드 품질을 높이는 수정을 명확히 분리할 필요가 있었다.

### AI 활용 & 검증

- AI가 범위 외 기능까지 구현하거나 과도한 설계를 제안하는 경우가 있어 지속적으로 범위를 통제해야 했다.
- AI가 생성한 결과를 그대로 사용하는 것이 아니라, PRD와 TEST_PLAN에 맞는지 지속적으로 검증해야 한다는 점을 체감했다.

## Try

### 시작 전 — 문제 정의 & 요구사항

- 다음 실습에서는 구현을 시작하기 전에 Mom Test를 먼저 수행하여 문제 정의와 요구사항 범위를 명확히 한 후 PRD를 작성하겠다.
- README를 PRD로 옮기는 과정에서 시간을 많이 사용했기 때문에, Requirement ID(FR/NFR/EXT)를 먼저 정의하고 세부 요구사항을 작성하는 방식을 적용해보겠다.

### RED & GREEN — TDD 사이클

- RED 단계에서는 Skeleton RED와 Executable RED를 명확히 구분하고, GREEN 진입 조건을 사전에 정의하여 혼란을 줄이겠다.
- GREEN 단계에서는 여러 테스트를 동시에 처리하기보다 RED ID 하나를 완료한 후 다음 단계로 진행하는 방식으로 TDD 사이클을 더 충실히 따르겠다.

### 설계 & 리팩터링

- ECB 적용 시 구현 전에 "이 책임은 Boundary, Control, Entity 중 어디에 속하는가?"를 먼저 판단하는 체크리스트를 만들어 활용하겠다.
- Golden Master는 출력 포맷 검증에만 사용하고, 비즈니스 로직 검증은 일반 테스트가 담당하도록 역할을 명확히 구분하겠다.
- 리팩터링 단계에서는 Change Budget을 먼저 정의한 후 작은 범위의 구조 개선만 수행하여 기능 변경이 섞이지 않도록 하겠다.

### AI & 개발 환경

- AI에게 구현을 요청하기 전에 범위(Scope), 완료 조건(DoD), 제외 대상(Out of Scope)을 명확히 전달하여 과도한 설계나 범위 확장을 줄이겠다.
- AI가 생성한 결과를 그대로 수용하기보다, PRD와 TEST_PLAN을 기준으로 검증하는 습관을 지속적으로 유지하겠다.
- Command, Skill, Rule, Hook의 역할을 잘 숙지하여 다음 실습에서 빠르게 환경을 구성할 수 있도록 하겠다.
