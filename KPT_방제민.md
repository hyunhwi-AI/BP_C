# UnitConverter 팀 프로젝트 KPT 회고

## 1. 회고 목적

이번 팀 프로젝트에서는 UnitConverter를 단순 구현이 아니라, PRD → 테스트 → 코드 → 리팩토링까지 추적 가능한 C2C 방식으로 진행했다.
Mom Test, R-G-I-O, PRD, Harness, Rule, Command, RED/GREEN/REFACTOR, Golden Master를 순서대로 적용하면서 Cursor AI를 단순 코드 생성 도구가 아니라 개발 프로세스를 지키는 협업 도구로 활용했다.

---

## 2. Check-In

각자 현재 상태를 1~5점으로 공유한다.

* 1점: 매우 힘듦
* 2점: 조금 힘듦
* 3점: 보통
* 4점: 좋음
* 5점: 최고

> 4점, 그 동안 바이브 코딩으로만 진행하다가 이처럼 추적가능한 방법론을 통하여 개발 진행해본 것이 좋은 경험이었습니다.
다만, 처음 접하는 내용들이 존재하여 시간이 조금 짧았던 것 가습니다.

## 4. Problem — 어려웠거나 개선이 필요한 점

| 항목   | 내용                                                                                                                   |
| ---- | -------------------------------------------------------------------------------------------------------------------- |
| P-01 | 워크북 흐름과 일반적인 브랜치 전략이 섞이면서 “PRD를 main에 먼저 넣는지, spec 브랜치에서 만드는지” 혼란이 있었다.                                              |
| P-02 | `.cursorrules`와 `.cursor/rules/*.mdc` 중 어떤 방식을 써야 하는지 초반에 판단이 어려웠다.                                                  |
| P-03 | Command가 많아지면서 `red-test-plan`, `red-skeleton`, `green-minimal`, `golden-master`, `refactor-safe`의 역할 구분이 헷갈릴 수 있었다. |
| P-04 | 처음에는 Command의 허용 파일을 너무 구체적으로 고정해서, 새 파일이나 새 unit 추가 시 Command를 다시 고쳐야 하는 문제가 있었다.                                   |
| P-05 | Golden Master 후보를 한 번에 모두 승인할지, 일부만 승인할지 판단이 어려웠다.                                                                   |
| P-06 | `/refactor-safe` 실행 시 Command 안에 이미 있는 규칙을 다시 길게 입력해야 하는지 헷갈렸다.                                                      |
| P-07 | 리팩토링 단계에서 기능 추가, 버그 수정, 구조 개선의 경계가 모호해질 수 있었다.                                                                       |
| P-08 | Architecture 테스트가 아직 부족해 SRP/OCP 위반을 테스트로 직접 보호하는 수준은 충분하지 않았다.                                                      |
| P-09 | 시간 부족으로 팀원의 수정사항을 충분히 리뷰하지 못했고, 내 수정사항에 대해 팀원이 준 리뷰 피드백에도 충분히 대응하지 못했다.       |
| P-10 | 리뷰를 언제, 어떤 기준으로, 어느 범위까지 해야 하는지에 대한 팀 그라운드룰이 없어 리뷰 우선순위가 뒤로 밀렸다.              |
| P-11 | 코드 변경은 있었지만 PRD → 테스트 → 코드 → 리뷰 관점에서 서로 확인하는 시간이 부족해 C2C 추적성을 팀 단위로 검증하지 못했다. |

## 5. Try — 다음에 시도할 개선 행동

| 항목   | 실행할 개선 행동                                                                                   |
| ---- | ------------------------------------------------------------------------------------------- |
| T-01 | 다음 프로젝트에서는 시작 전에 “워크북 기준 브랜치 순서”와 “실무형 브랜치 순서”를 구분해 팀 공통 기준으로 합의한다.                         |
| T-02 | Command를 만들 때 파일명 고정 방식보다 “계층/책임 기준 허용 범위”로 작성한다.                                           |
| T-03 | 각 Command의 목적을 한 줄로 정리한 `docs/COMMAND_MAP.md`를 만들어 팀원이 헷갈리지 않도록 한다.                         |
| T-04 | Golden Master는 처음 2~3개로 구조를 검증한 뒤, 나머지를 추가하는 방식으로 진행한다.                                     |
| T-05 | `/refactor-safe` 실행 시에는 긴 프롬프트를 반복하지 않고 `Smell ID + 목표 + Budget`만 입력하는 팀 규칙을 둔다.            |
| T-06 | 리팩토링 전에는 반드시 `/refactor-smell`을 먼저 실행하고, 한 번에 스멜 1개만 처리한다.                                  |
| T-07 | 다음 반복에서는 `T-ARCH-SRP-01`, `T-ARCH-OCP-01` 같은 Architecture 테스트를 추가해 구조 품질도 테스트로 보호한다.        |
| T-08 | 추가 요구사항인 설정 파일 로드, 동적 단위 등록, JSON/CSV/table 출력 포맷은 각각 별도의 RED → GREEN → REFACTOR 사이클로 진행한다. |
| T-09 | 다음 프로젝트에서는 작업 시작 전에 Code Review Ground Rules를 먼저 정한다.                                                                                     |
| T-10 | 리뷰 요청 시 “변경 목적, 관련 PRD/테스트 ID, 확인해야 할 파일, 리뷰 요청 범위”를 함께 남긴다.                                                                              |
| T-11 | 각 단계별로 리뷰 타임박스를 둔다. 예: RED 리뷰 5분, GREEN 리뷰 10분, REFACTOR 리뷰 10분.                                                                          |
| T-12 | 리뷰어는 모든 코드를 다 보려 하지 말고, 이번 브랜치의 목적에 맞는 핵심만 확인한다. 예: RED에서는 테스트가 실패하는지, GREEN에서는 테스트 기대값을 바꾸지 않았는지, REFACTOR에서는 Golden Master가 유지되는지 확인한다. |
| T-13 | 리뷰 피드백은 “반드시 반영 / 논의 필요 / 다음 사이클로 보류” 세 가지로 분류해 처리한다.                                                                                     |

## 7. 최종 회고 요약

이번 프로젝트에서 가장 잘 된 점은 AI에게 바로 구현을 맡기지 않고, Mom Test와 PRD, Rule, Command, Test Loop를 먼저 구성한 것이다. 이 덕분에 Cursor AI가 임의로 코드를 확장하거나 테스트를 약화하지 않도록 통제할 수 있었다.

가장 어려웠던 점은 워크북의 실습 흐름과 실제 프로젝트 브랜치 전략, Command 역할을 구분하는 것이었다. 특히 `red-test-plan`, `red-skeleton`, `green-minimal`, `golden-master`, `refactor-safe`의 역할을 명확히 하지 않으면 작업 단계가 섞일 수 있었다.

다음 프로젝트에서는 초반에 브랜치 전략과 Command Map을 먼저 합의하고, 각 단계마다 “지금은 RED인지, GREEN인지, REFACTOR인지”를 명시한 뒤 작업한다. 또한 구조 품질을 더 잘 보호하기 위해 Architecture 테스트를 추가한다.
